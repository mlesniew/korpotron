# Core Data Model — Plan Brief

> Full plan: `context/changes/core-data-model/plan.md`

## What & Why

Define the three ORM models that underpin every user-facing feature in Korpotron: `Template`, `OptionGroup`, and `Option`. Without these tables, S-01 (template management), S-02 (option group management), and S-03 (text generation) cannot be built. This is a pure data-layer foundation slice — no UI, no API, just the schema.

## Starting Point

No custom Django apps or models exist. `INSTALLED_APPS` contains only built-in Django apps (`settings.py:34-41`). The `korpotron/` package is configuration-only (settings, URL routing, WSGI). This plan creates the first app and the first custom migration.

## Desired End State

A `core` Django app is registered and its migration is applied. Three tables exist in the database. Django admin shows all three models with useful column displays for development inspection. Pytest tests confirm field storage, `__str__`, and FK cascade deletion. Docker build passes.

## Key Decisions Made

| Decision | Choice | Why (1 sentence) |
|---|---|---|
| App placement | New `core` app | Keeps `korpotron/` as project-config-only; standard Django separation of concerns. |
| `is_response` flag | Deferred to v2 | PRD already designated it nice-to-have; keeping the schema minimal for now. |
| Option ordering | pk order (no `order` field) | MVP option sets are small and manually reordering is not needed before S-02. |
| Admin registration | Yes, with `list_display` | Free dev-time inspection; 15 lines of cost. |

## Scope

**In scope:**
- `core` Django app scaffold (`__init__.py`, `apps.py`, `models.py`, `admin.py`, `migrations/`)
- `Template`, `OptionGroup`, `Option` models with FKs and `__str__`
- `INSTALLED_APPS` update in `settings.py`
- Initial migration generated and applied
- Pytest tests for creation, `__str__`, and cascade deletion

**Out of scope:**
- `is_response` field on `Template` (v2)
- Option display ordering field
- Template–OptionGroup explicit association (handled at query time in S-03)
- Any views, forms, or URL routing

## Architecture / Approach

Three models in a single `core/models.py`. `Template` and `OptionGroup` each carry a direct FK to `AUTH_USER_MODEL` (cascade delete). `Option` carries a FK to `OptionGroup` (cascade delete). No M2M relations, no through tables. Admin registrations in `core/admin.py` using the `@admin.register` decorator pattern. Migration generated by Django's built-in `makemigrations`.

## Phases at a Glance

| Phase | What it delivers | Key risk |
|---|---|---|
| 1. Core app — models, admin, migration | App scaffold, all three models, admin, migration applied | Field names or `max_length` choices that are hard to change once S-01/S-02 depend on them |
| 2. Tests | pytest coverage for creation, `__str__`, cascade + Docker build verified | None significant |

**Prerequisites:** `SECRET_KEY` set in `.env` (already done post auth-scaffold)  
**Estimated effort:** ~1 session across 2 phases

## Open Risks & Assumptions

- `max_length=200` for name fields is assumed sufficient — can be raised in a later migration if needed
- No index on `Template.user` or `OptionGroup.user` beyond the implicit FK index; acceptable at small scale

## Success Criteria (Summary)

- `uv run manage.py migrate` applies cleanly and `uv run manage.py check` passes
- All three models appear and are usable in Django admin
- `uv run pytest` passes (existing auth tests + new model tests) and `docker build .` succeeds
